Concepto 1

    
¿Qué es un programa?

    
Un programa es una lista ordenada de instrucciones que una máquina puede ejecutar. Nada más.


    

      

        
Si solo se quedan con una idea

        
La CPU no entiende Python. Python le traduce.

        
Aquí no buscamos memorizar hardware, sino fijar un mapa: texto legible arriba, señales eléctricas abajo, y un traductor entre ambos.

      

      

        

          CPU
          Trabaja con señales muy primitivas, no con palabras como print o if.
        

        

          Lenguaje
          Es la capa humana: nos permite escribir instrucciones que podamos leer y razonar.
        

        

          Traductor
          CPython convierte ese texto en algo que finalmente termina ejecutando la máquina.
        

      

    


    
La CPU de tu ordenador — el chip físico — no entiende español, ni Python, ni nada legible. Solo entiende corriente eléctrica: voltaje alto o voltaje bajo. Eso es un 1 o un 0.


    
Todo lo que ocurre en un ordenador se reduce a largas cadenas de unos y ceros que representan instrucciones primitivas: «mueve este dato de aquí a allí», «suma estos dos números», «escribe en pantalla».


    
Escribir eso a mano es imposible. Por eso existen los lenguajes de programación: una forma de escribir instrucciones que tú puedas leer. Pero alguien tiene que hacer la traducción.


    

      

        
Tú escribes

        
print("hola")

      

      
→

      

        
Traductor

        
Python

      

      
→

      

        
CPU ejecuta

        
01101001...

      

    


    

      

        
1. Token

        
print deja de ser solo texto y pasa a ser una pieza reconocible del lenguaje: un identificador que puede referirse a una built-in.

      

      

        
2. AST

        
El parser ve print("hola") y construye una estructura del tipo: llamada a función + argumento string. Eso ya es sintaxis, no caracteres.

      

      

        
3. CPU

        
Cuando la VM ejecuta ese nodo, termina invocando código nativo del runtime. Ese binario es el que la CPU entiende y ejecuta de verdad.

      

    


    

      📌 En resumen
      Un lenguaje de programación es una forma de escribir instrucciones legibles para humanos. El «traductor» convierte esas instrucciones a algo que la CPU puede ejecutar. Python es uno de esos traductores.
    


    

      
Pruébalo

      
👁 Dos formas de «leer» el mismo código

      
Escribe código Python y compara cómo lo lee la CPU y cómo lo lee Python:

      
    


    

      

        

          
💡 La CPU ve: caracteres sueltos

          
Cada letra es un número binario independiente (ASCII). La p, la r, la i... no significan nada juntas.

          

        

        

          
🐍 Python ve: tokens con significado

          
print no son 5 letras sueltas: el tokenizador lo reconoce como un nombre especial del lenguaje y el parser entiende que aquí hay una llamada a función. "hola" pasa a ser un string.

          

        

      

    


    

      💡 ¿Ves la diferencia?
      La CPU ve una cadena plana de bits agrupados en bytes. Python, en cambio, agrupa caracteres en piezas con significado: nombres, operadores, paréntesis, strings. Eso es lo que hace el tokenizador — el primer paso del pipeline.
      

        

          
Qué pasa con print

          
Primero aparece como texto. Luego CPython lo trocea, el parser detecta que va seguido de paréntesis y construye un nodo AST de llamada. En ejecución, la VM resuelve el nombre print contra la tabla de built-ins.

        

        

          
Qué hay detrás

          
La función built-in print vive en el runtime de CPython, escrito sobre todo en C. Ese código C acaba disparando instrucciones máquina y llamadas al sistema para escribir bytes en la terminal.

        

      

    

  


  


  
  

    
Concepto 2

    
Python es, él mismo, un programa

    
Esta es la pieza que hace que todo empiece a encajar.


    

      

        
La confusión que quiero evitar

        
Lenguaje y ejecutable no son lo mismo

        
Los alumnos suelen mezclar «Python el lenguaje» con «Python el programa instalado». Esta sección separa esas dos cosas.

      

      

        

          Tu .py
          Es solo una receta escrita. Por sí mismo no ejecuta nada.
        

        

          CPython
          Es el ejecutable que sabe leer esa receta y coordinar la ejecución.
        

        

          Instalar Python
          Significa instalar ese intérprete, no solo aprender una sintaxis.
        

      

    


    
Cuando instalas Python en tu ordenador, lo que instalas no es «un lenguaje». Instalas un ejecutable: python.exe en Windows, python3 en Mac/Linux. Ese ejecutable ocupa espacio en tu disco, tiene bytes, es un binario. Se llama CPython y está escrito en el lenguaje C.


    

      

        
        
Tu fichero

        
hola.py — Texto plano. Cualquiera puede leerlo. No hace nada por sí solo.

      

      

        
        
El intérprete

        
python.exe — Un programa que sabe leer tu fichero y decirle a la CPU qué hacer.

      

    


    
Es como un PDF: el fichero .pdf no hace nada solo. Necesitas Adobe Reader (un programa) para abrirlo. Tu .py necesita a python.exe.


    

      

        
📕 Compilado — C, Rust

        
Traduce todo tu código antes de ejecutar. Genera un binario (.exe). Más rápido, pero el ciclo de desarrollo es lento.

        

          gcc programa.c -o programa
        

      

      

        
🗣️ Interpretado — Python

        
Lee y ejecuta al momento. Sin paso previo. Si algo falla, te avisa inmediatamente en la línea exacta.

        

          python programa.py
        

      

    


    

      💡 Momento AHA
      Python (el programa) es un chef. Tu fichero .py es la receta. El chef lee la receta paso a paso y cocina. Sin chef, la receta es solo un papel.
    

  


  


  
  

    
Concepto 3

    
Qué hace CPython por dentro

    
Cuando ejecutas python hola.py, CPython recorre cinco etapas muy concretas. No son cajas mágicas: cada una transforma el programa un poco más.


    

      

        
Objetivo docente

        
Usar el pipeline como mapa de depuración mental

        
No hace falta que memoricen cada caja. Sí quiero que sepan distinguir entre entender estructura, preparar ejecución y fallar en runtime.

      

      

        

          Pasos 1 y 2
          CPython intenta entender la forma del programa. Aquí viven los problemas de sintaxis.
        

        

          Pasos 3 y 4
          Prepara y ejecuta. La sintaxis puede estar bien y aun así fallar en este punto.
        

        

          Regla útil
          SyntaxError suele significar «no entiendo la frase»; NameError, «la frase se entiende, pero falla al hacerla».
        

      

    


    

      

        
1

        
Tokenización

        
Rompe el texto en unidades léxicas: nombres, operadores, paréntesis, strings. Aquí todavía no se “entiende” el programa completo.

      

      

        
2

        
Parsing → AST

        
El parser intenta encajar esos tokens en la gramática de Python. Si no puede construir un árbol sintáctico válido, lanza SyntaxError.

      

      

        
3

        
Bytecode

        
El AST se compila a bytecode: instrucciones para la VM de Python, no para la CPU. Puede guardarse como .pyc en __pycache__/.

      

      

        
4

        
Python VM

        
La VM vive dentro del proceso python.exe o python3. Ejecuta el bytecode instrucción por instrucción y aquí aparecen NameError o TypeError.

      

      

        
5

        
Output

        
La ejecución produce efectos reales: texto en pantalla, ficheros, llamadas a red, cálculos, gráficos o lo que tu programa haya pedido.

      

    


    

      

        
Tu fichero

        
.py

      

      
→

      

        
Tokens

        
NAME ( STRING )

      

      
→

      

        
AST

        
🌳

      

      
→

      

        
Bytecode

        
.pyc

      

      
→

      

        
VM

        
python.exe

      

      
→

      

        
Output

        
hola

      

    


    

      

        
🚨 SyntaxError

        
Falla en el paso 2. El parser no logra construir el AST porque la secuencia de tokens no respeta la gramática del lenguaje.

      

      

        
⚠️ NameError / TypeError

        
Falla en el paso 4. La sintaxis era correcta, pero en ejecución algo no cuadra: falta un nombre, se divide entre cero o mezclas tipos incompatibles.

      

    


    

      ⚠️ Importante
      
CPython no compila directamente a código máquina nativo para tu CPU. Compila a bytecode, que es un lenguaje intermedio para su VM. El código binario es lo que ejecuta la CPU; el bytecode lo ejecuta la VM que va dentro de python.exe.

      

        

          
AST en una frase

          
Es un árbol que representa la estructura gramatical del programa. Por ejemplo, “llamada a función” con nombre print y argumento string "hola".

        

        

          
Por qué detecta sintaxis

          
Porque el parser intenta casar los tokens con las reglas del lenguaje. Si falta un paréntesis, una coma o un nombre válido, no puede completar el árbol y se detiene.

Concepto 4

    
El REPL y Jupyter

    
CPython también funciona sin fichero. Puede leer una línea, ejecutarla, devolverte el resultado, y esperar la siguiente.


    

      

        
Lo importante aquí

        
Jupyter no es otra cosa distinta: es el REPL convertido en espacio de trabajo

        
Esta transición es clave para el curso: del terminal pasamos a un entorno donde pensar, probar, explicar y simular en la misma pantalla.

      

      

        

          REPL
          Conversación directa con Python: escribes, ejecuta, responde, espera la siguiente línea.
        

        

          Notebook
          Esa misma conversación, pero troceada en celdas, con texto y resultados visibles.
        

        

          Kernel
          Es la memoria compartida que mantiene variables, imports y funciones entre celdas.
        

      

    


    
Se llama REPL: Read → Eval → Print → Loop. Es una conversación:


    

      

        

        python (REPL)
      

      
>>> 2 + 2
4
>>> nombre = "Ana"
>>> print(f"Hola {nombre}")
Hola Ana
>>> # la variable sigue en memoria
>>> len(nombre)
3

      🚀 Ahora entra Jupyter
      
El REPL es una conversación en línea. Jupyter Notebook convierte esa misma conversación en un laboratorio visual: celdas, resultados, gráficos, texto explicativo y estado compartido por un mismo kernel.

Por qué importa

          
En un notebook puedes mezclar semántica, sintaxis, código ejecutable y resultados sin salir de la misma pantalla. Para una primera clase, eso es oro.

        

        

          
Qué mantiene el kernel

          
Variables, imports, funciones y estado. Si defines spread en una celda, puedes usarlo en la siguiente. Si reinicias el kernel, desaparece todo.

        

        

          
Por qué encaja en el curso

          
Es el formato perfecto para pasar de intuición a simulación: explicas una idea, ejecutas una celda, cambias un parámetro y enseñas el efecto al instante.

        

      

    


    

      

        
        
Ideal para explorar

        
Calculas algo, ves el resultado, decides el siguiente paso. Es como hacer química en un laboratorio, no en una fábrica.

      

      

        
        
Las variables viven

        
Lo que defines en la celda 1 existe en la celda 5. No empiezas de cero cada vez. El kernel mantiene el estado.
